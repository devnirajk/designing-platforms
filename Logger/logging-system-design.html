Logging System - Low Level Design

A thread-safe, extensible logging framework with async I/O, level-based filtering, and pluggable sinks.

1. Functional Requirements

Core Logging

  • Log messages with timestamp, level, TAG (class name), and message
  • Support 5 levels: DEBUG, INFO, WARN, ERROR, FATAL
  • Level-based filtering at initialization (e.g., set to WARN = only WARN, ERROR, FATAL pass through)

Multiple Sinks

  • Logs write to both file and console
  • Sinks are pluggable via the LogSink interface
  • Adding a new destination (DB, remote server) requires zero changes to existing code

Thread Safety

  • Multiple threads can log simultaneously
  • Caller thread should return immediately (non-blocking)
  • Actual writing happens on a separate consumer thread

Design Constraints

  • Singleton Logger instance (thread-safe double-checked locking)
  • Queue-based decoupling between producer and consumer
  • No shared mutable state between caller threads

3. Design Patterns Used

Singleton Logger

Only one Logger instance exists across the entire application. Uses double-checked locking with synchronized(Logger.class) for thread-safe lazy initialization.

Why? Multiple Logger instances would mean multiple queues, inconsistent filtering, and resource duplication.

if(INSTANCE == null) {
    synchronized(Logger.class) {
        if(INSTANCE == null) {
            INSTANCE = new Logger();
        }
    }
}

Strategy LogSink Interface

Each destination implements LogSink.log() differently. The dispatcher doesn't care how a sink writes -- it just calls the interface method.

Why? Open/Closed Principle. Adding a new sink (e.g., DatabaseLogger) requires zero changes to Logger or Dispatcher.

interface LogSink {
    void log(LogMessage message);
}

Pub-Sub Dispatcher Fan-out

LogDispatcher takes one message and fans it out to all registered sinks. Sinks don't know about each other or the queue.

Why? Decouples producers from consumers. The Logger doesn't need to know how many sinks exist or what they do.

for(LogSink sink : list) {
    sink.log(message);
}

Producer-Consumer Queue

Caller threads produce messages into a LinkedBlockingQueue. The dispatcher thread consumes them asynchronously.

Why? Decouples the speed of logging from the speed of writing. Callers never block on I/O.

// Producer (any thread)
queue.add(logMessage);

// Consumer (dispatcher thread)
LogMessage msg = queue.take();

4. Queue Choice: Why LinkedBlockingQueue?

We evaluated three queue types before settling on LinkedBlockingQueue:

Property LinkedList / ArrayDeque ConcurrentLinkedQueue LinkedBlockingQueue
Thread-safe? No Yes (lock-free) Yes (lock-based)
Blocking take()? No No -- poll() returns null Yes -- take() blocks until available
Consumer behavior when empty Crashes or busy-loops Busy-loops (wastes CPU) Sleeps efficiently (no CPU waste)
Bounded capacity? No No Optional (can set max size)
Best for Single-threaded use High-throughput, non-blocking reads Producer-consumer with blocking

The Key Insight

The consumer thread needs to wait efficiently when no logs are available. With ConcurrentLinkedQueue, the consumer would need a while(true) { poll(); } busy-loop, wasting CPU cycles. LinkedBlockingQueue.take() uses OS-level wait/notify -- the thread sleeps with zero CPU usage until a producer adds a message.

Additionally, LinkedBlockingQueue supports an optional bounded capacity, which acts as backpressure: if the queue fills up, producers block on put() instead of producing infinitely and causing an OOM.

5. Class-by-Class Breakdown

LogLevel.java

Enum with numeric priority. Lower number = higher severity. Used by Logger.filter() to compare against the configured threshold.

enum LogLevel {
    DEBUG(5), INFO(4), WARN(3), ERROR(2), FATAL(1);

    public final int level;
    LogLevel(int level) { this.level = level; }
}

Filtering logic: If logLevelFilter = WARN (3), then only levels with level <= 3 pass through (WARN, ERROR, FATAL). DEBUG(5) and INFO(4) are filtered out because 3 >= 5 is false.

LogMessage.java

Immutable data object carrying all log metadata. Created fresh per log call -- no shared state.

class LogMessage {
    private LogLevel logLevel;
    private String message;
    private String TAG;          // Source class name
    private LocalDateTime time;  // Captured at creation
    // constructor + getters
}

Logger.java

Singleton entry point. Provides static methods d(), i(), w(), e(), f(). Each method:

  1. Checks the level filter
  2. Creates a LogMessage locally (no shared field)
  3. Adds it to the LinkedBlockingQueue
public static void d(String message, String TAG) {
    if(filter(LogLevel.DEBUG)) {
        addToMessageQueue(new LogMessage(LogLevel.DEBUG, message, TAG));
    }
}

Key decision: No synchronized on log methods. The queue is thread-safe, and LogMessage is created locally per call -- no contention.

LogSink.java

Single-method interface. Each destination implements this differently.

interface LogSink {
    void log(LogMessage message);
}

FileLogger.java

Writes to a file using BufferedWriter with try-with-resources (auto-close). Opens in append mode.

class FileLogger implements LogSink {
    private File file;
    @Override
    public void log(LogMessage message) {
        try (BufferedWriter writer = new BufferedWriter(
                new FileWriter(file, true))) {
            writer.write(/* formatted message */);
            writer.newLine();
            writer.flush();
        }
    }
}

ConsoleLogger.java

Simplest sink -- just prints to stdout.

class ConsoleLogger implements LogSink {
    @Override
    public void log(LogMessage message) {
        System.out.println(/* formatted message */);
    }
}

LogDispatcher.java

The heart of the system. Owns the consumer thread, takes from the queue, fans out to all sinks.

class LogDispatcher {
    private LinkedBlockingQueue<LogMessage> queue;
    private ExecutorService service;
    List<LogSink> list;

    LogDispatcher(Logger logger, ExecutorService service, List<LogSink> list) {
        this.queue = logger.getQueue();
        this.service = service;
        service.submit(() -> consumer());
        this.list = list;
    }

    public void consumer() {
        while(true) {
            LogMessage message = queue.take();  // blocks until available
            for(LogSink sink : list) {
                try {
                    sink.log(message);
                } catch(Exception e) {
                    System.out.println(e.getMessage());
                }
            }
        }
    }
}

Key decisions:

  • take() blocks efficiently -- no CPU waste when idle
  • Try-catch inside the for-loop so one failing sink doesn't skip others
  • InterruptedException triggers graceful shutdown

6. End-to-End Flow

1
Application calls Logger.d("User logged in", "AuthService")

Any thread, any class. Static method -- no instance needed after init.

2
Logger.filter() checks level

If logLevelFilter = WARN(3) and DEBUG = 5, then 3 >= 5 is false. Message is dropped. No object creation, no queue overhead.

3
If passes filter: create LogMessage locally

new LogMessage(LogLevel.DEBUG, message, TAG) -- captures timestamp at this moment. Stack-local, no shared state.

4
Add to LinkedBlockingQueue

Thread-safe queue.add(). Caller thread returns immediately. Zero I/O on the caller's thread.

5
Dispatcher's consumer thread wakes up

queue.take() was blocking -- now it returns the message. Runs on a separate thread in the ExecutorService.

6
Fan-out to all registered sinks

Iterates over List<LogSink> -- calls sink.log(message) on each. FileLogger writes to disk, ConsoleLogger prints to stdout.

8. Trade-offs & Interview Talking Points

Why not synchronize log methods?

Initially we had synchronized on d(), i(), etc. This meant every caller thread would contend on the same lock even though they have no shared state. By creating LogMessage locally and using a thread-safe queue, we eliminated the need for any lock on the hot path.

Why not one queue per sink?

Multiple queues means the Logger must know about every sink. Adding a new sink requires changing Logger. The dispatcher pattern gives us a single queue with fan-out -- Logger stays unchanged regardless of how many sinks exist.

Single consumer thread limitation

Currently one thread dispatches to all sinks sequentially. If FileLogger is slow, ConsoleLogger waits. For production, you could dispatch to sinks in parallel using the ExecutorService, or give each sink its own queue + consumer.

File open/close per message

FileLogger opens a new BufferedWriter for each log line (try-with-resources). This is safe but slow for high throughput. An optimization: keep the writer open as a field and flush periodically, or use a FileChannel with memory-mapped I/O.

Unbounded queue risk

LinkedBlockingQueue without a capacity is unbounded. If producers are faster than consumers, the queue grows indefinitely causing OOM. Fix: pass a capacity like new LinkedBlockingQueue<>(10000) and use put() to apply backpressure.

Graceful shutdown

The while(true) consumer loop needs a clean exit path. When InterruptedException is caught (from take()), the service shuts down. In production, you'd also want to drain the queue before exiting so no logs are lost.

Bootstrap Example (Main.java)

public class Main {
    public static void main(String[] args) {
        // 1. Create shared queue
        LinkedBlockingQueue<LogMessage> queue = new LinkedBlockingQueue<>();

        // 2. Initialize singleton logger
        Logger logger = Logger.instance(LogLevel.WARN, queue);

        // 3. Create sinks
        List<LogSink> sinks = List.of(
            new ConsoleLogger(),
            new FileLogger("/var/log/app.log")
        );

        // 4. Start dispatcher (consumer thread begins immediately)
        ExecutorService service = Executors.newSingleThreadExecutor();
        new LogDispatcher(logger, service, sinks);

        // 5. Use from anywhere
        Logger.w("Connection timeout", "HttpClient");   // passes filter
        Logger.d("Cache hit ratio: 92%", "CacheService"); // filtered out
        Logger.e("Null pointer in handler", "ApiGateway"); // passes filter
    }
}
